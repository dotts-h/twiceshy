# Test hardening plan

This is an adversarial test audit for the engine repo and the sibling corpus repo.
It is intentionally an action list: each row names the missing invariant, the exact
test to add, and why the current suite missed the failure.

## A. Engine invariants and property tests

| Priority | Severity | Missing invariant | Exact test to add | Why current tests miss it |
| --- | --- | --- | --- | --- |
| A1 | critical | ID allocation must be merge-safe, not just locally max+1: two independently green branches cannot introduce the same `exp-NNNN` id. | `internal/ingest`: `TestNextID_RebasesAgainstCurrentMainBeforePR` once the allocation/CI helper exists; fixture creates a base corpus, branch A with `exp-2760`, branch B allocated from stale base, then asserts branch B's merge-time check rejects or reallocates before PR. Also add `internal/record`: `TestLoadCorpus_DuplicateIDsAcrossDistinctPaths` using two valid records in different files with the same frontmatter `id`. | `internal/index.TestNextID`, `internal/ingest.TestNextID`, and `internal/record.TestMaxID` only prove max+1 within one observed tree or stale index-vs-disk. They do not simulate two PR heads that were valid separately but invalid after both merge. Strict `record.LoadCorpus` detects the final bad state, but too late: after the corpus main branch is already poisoned. |
| A2 | critical | Every write path must allocate through `ingest.NextID(ctx, ix, corpusRoot)`, never raw `index.NextID(ctx)`, before creating a persisted/proposed record. | `internal/server`: `TestRecordAndReportAllocationUseSharedReservedIDs` extends `TestConcurrentRecordExperienceAllocatesDistinctIDs` and `internal/server/report_test.go`'s concurrent report test with mixed `record_experience` + `report_outcome` calls. Assert returned markdown ids are strictly unique and monotonic across both tools. | Existing server concurrency tests cover separate paths, but not a mixed workload. `index.NextID` is still public inside the module and documented as non-atomic, so a future path can regress to stale max+1 without a single cross-tool guard failing. |
| A3 | high | Schema-version behavior must be strict on write/CI and tolerant only on serve; unsupported `schema_version` must be reported as critical, not silently skipped. | `internal/record`: `TestSchemaVersion_StrictVsServeModeMatrix`; table covers `Parse`, `ParseLenient`, `LoadCorpus`, `LoadCorpusResilient`, and `LoadCorpusForServe` for current, additive unknown field, and future schema version. Assert only `LoadCorpusForServe` returns `critical` for future schema. | `internal/record/schema_test.go` checks marshal vs JSON Schema; `cmd/twiceshy/main.go` uses tolerant serve logic, but no single matrix locks the intended split. This is exactly the kind of scope confusion that created exp-0746-style outages. |
| A4 | high | State-machine transitions must be closed: only defined transitions may mutate lifecycle fields, and invalid combinations are rejected. | `internal/promote`: `TestLifecycleStateMachineTransitions`; table over `quarantined -> validated`, `validated -> stale`, `validated -> disputed`, stale/disputed `repromote -> validated`, and invalid transitions. Assert status plus `validated_at`, `promotion`, `demotion`, and `valid.until` invariants. | `internal/promote/promote_test.go`, `internal/promote/adapt_test.go`, `cmd/twiceshy/repromote_test.go`, and `internal/record/provenance_desync_test.go` cover examples, not the transition graph as a property. A new status or provenance field can drift one path without tripping all examples. |
| A5 | high | Promote throttle and hold cooldown must keep throughput bounded without committing fixed operational paths into every validate PR. | `cmd/twiceshy`: `TestPromoteEffect_CommitsOnlyRecordDeltasAndPerRunManifest`; run promote in a temp git corpus, then assert `git diff --name-only --cached` contains only `experience/...` and `runs/run-*-promote.json`, never `runs/promote.holds.json` or a fixed journal name. | `cmd/twiceshy/holds_test.go` validates ledger semantics and path, but not git staging. `scripts/scheduled-validate.sh` currently runs `git add -A`, so fixed `runs/promote.holds.json` can be committed and collide across validate PRs. |
| A6 | high | Dedup-at-ingest must be invariant across all signals, orderings, statuses, and import reruns. | `internal/ingest`: `FuzzPrepareDedupSignals`; generate drafts with permutations of signatures/summary/title and an index containing validated and quarantined matches. Assert any exact signature match returns `Known`, importer mode with `IncludeQuarantined` is idempotent, and non-matches always produce `Status=quarantined`. | `internal/ingest/prepare_test.go` has good examples, including non-first signature and `IncludeQuarantined`, but not generated permutations. A future probe reorder or early return could reintroduce duplicate imports. |
| A7 | high | Retrieval hard bounds must hold for every query: `len(hits) <= index.MaxK`, floor semantics are consistent, and quarantined records never reach push. | `internal/index`: `FuzzRetrieveInvariants`; seed mixed validated/quarantined records, arbitrary hostile query text, stack filters, and `K` values. Assert cap, no errors, no quarantined unless `IncludeQuarantined`, fingerprint hits precede lexical hits, and `RetrievePush` never serves quarantine. | `internal/index/fuzz_test.go` only proves FTS input safety. `retrievepush_livecorpus_test.go` is behind the `livecorpus` tag and therefore absent from `make test`. Push safety should have a fixture-level non-live guard. |
| A8 | medium | `record.LoadCorpus` structural checks should be generated over small corpora: duplicate ids, dangling `superseded_by`, missing repros, bad paths, and sorted output. | `internal/record`: `FuzzLoadCorpusStructuralIntegrity`; generate 1-6 record files plus repro files, then compare `LoadCorpus` result/error to an oracle over ids/references/repro paths. | Existing `LoadCorpus` checks are mostly exercised indirectly by live fixture loads. The duplicate-id production outage proves direct structural tests need adversarial combinations. |
| A9 | medium | `similarity.Assess` should preserve mathematical properties used for review leads: containment in `[0,1]`, identical draft/reference is 1 when enough words, no divide-by-zero, deterministic sorted matches. | `internal/similarity`: `FuzzAssessProperties`. | `internal/similarity` has examples, but the function is a guardrail signal. Cheap generated checks prevent review tooling from emitting unstable or impossible scores. |
| A10 | medium | Coverage floor must measure the failed seams, not just aggregate percentage. | Add a scriptable package gate in `Makefile`: `cover-critical`, parsing `go tool cover -func` and requiring non-zero/threshold coverage for `internal/ingest/nextid.go`, `internal/record/record.go`, `internal/index/index.go`, `internal/notify/notify.go`, `internal/promote/*.go`, `internal/run/*.go`, and `cmd/twiceshy/holds.go`. | `COVER_FLOOR=80` is gameable by broad easy coverage. It did not force shell ops tests and would not have required merge/auth/allocation invariants. |

## B. Corpus CI hard gate

| Priority | Severity | Missing invariant | Exact test/check to add | Why current tests miss it |
| --- | --- | --- | --- | --- |
| B1 | critical | A PR must be valid against the current target branch at merge time, including ids allocated after the PR branch was cut. | In `/home/ori/twiceshy-corpus/.forgejo/workflows/validate.yml`, add a step after checkout: fetch `origin/main`, compute new/modified `experience/YYYY/*.md` on the PR head, then run a pinned engine command such as `twiceshy corpus-merge-check -base origin/main -head HEAD -corpus .`. Test in engine as `cmd/twiceshy`: `TestCorpusMergeCheckRejectsCrossBranchDuplicateIDs`. Assert base has `exp-2759`, PR head adds another file whose frontmatter id is `exp-2759`, and the command fails before `index`. | The current corpus workflow runs `twiceshy index -corpus .` on the PR checkout only. Each stale branch can be internally unique and green. The duplicate appears only after another green PR lands first. |
| B2 | critical | Allocation must be validated against remote main, not only local disk. | Add a lightweight `twiceshy nextid -corpus . -base origin/main` or `twiceshy corpus-merge-check --allocation-only` command. Test `cmd/twiceshy`: `TestNextIDCommandUsesRemoteMainMax`; fixture has local max 2758 and fetched main max 2768, command must return `exp-2769`. | `ingest.NextID` uses local disk max. In a dedicated long-lived import/validate clone, local disk can be stale until fetch/reset, and the PR branch can be stale relative to remote main after another PR merges. |
| B3 | high | Corpus CI must reject fixed operational files in PRs, especially `runs/promote.holds.json` and any non-run-specific journal. | Add workflow step `twiceshy corpus-pr-paths -base origin/main -head HEAD`; engine test `cmd/twiceshy`: `TestCorpusPRPathsRejectFixedRunsFiles`. Allow only `experience/**`, `experience/repro/**`, and `runs/run-*-{promote,adapt}.json` if manifests remain committed. Reject `runs/promote.holds.json`, `runs/*-journal.json` without run id, `.twiceshy-loop.lock`, and generated DBs. | `record.LoadCorpus` ignores `runs/`; the current hard gate can be green while every validate PR carries a fixed path that conflicts with every other validate PR. |
| B4 | high | Doctors in corpus CI must never fail a PR based on quarantined records unless the check is explicitly a born-stale advisory pre-gate. | Keep `doctor staleness` report-only in workflow and add engine test `cmd/twiceshy`: `TestCorpusCIStalenessDoctorReportOnlyForQuarantine`, exercising the CLI exit code on a quarantined EOL record. | `internal/doctor/staleness_test.go` covers `Run` scoping, but the workflow contract is exit-code behavior. exp-0746 was a guard-scope outage, not a doctor algorithm panic. |
| B5 | medium | The pinned engine SHA in corpus CI must not silently lag schema/validator changes. | Add workflow check comparing `ENGINE_SHA` to an allowlisted engine release marker, plus engine-side test for a generated `docs/CORPUS_ENGINE_CONTRACT.md` if adopted. | Current `validate.yml` pins `ENGINE_SHA` by hand. A stale pin can green-light data the deployed server cannot read, or reject data the current engine accepts. |

## C. Shell ops layer

Recommended harness: use a Go-driven shell harness for high-value scripts, not ad hoc shell tests. A Go test can create temp repos, fake `git`, `curl`, `jq`, `forgejo-ci-merge`, and `systemctl` binaries on `PATH`, execute the script as a subprocess, and assert stdout/stderr, exit code, files, and captured HTTP headers. Keep the existing `.test.sh` smoke tests, but make the authoritative tests regular `go test` so `make test` runs them.

| Priority | Severity | Missing invariant | Exact test to add | Why current tests miss it |
| --- | --- | --- | --- | --- |
| C1 | critical | Merge result must not be swallowed: an import PR left open by `forgejo-ci-merge` must notify and exit with an observable status/log. | `scripts_test.go` or `internal/ops`: `TestScheduledImportMergeResultNotSwallowed`; fake `forgejo-ci-merge` returns 1 and fake curl records ntfy body. Assert notification contains `left OPEN` and PR number. | `scripts/scheduled-import.sh` has the behavior in comments/code, but no test executes it. Existing shell tests cover watchdogs and stall alarm, not import PR merge handling. |
| C2 | critical | Validate merge step must only merge green, soaked, non-anomalous `validate/*` PRs and must notify on merge failure. | `internal/ops`: `TestScheduledValidateMergeDueOnlyMergesGreenSoakedNonAnomaly`; fake Forgejo `/pulls` JSON with young, old, anomalous, non-validate, and failing merge cases. Assert only the eligible PR is passed to `forgejo-ci-merge`, anomalous PR is skipped, and failed merge notifies. | `scripts/scheduled-validate.sh` has no test. Its `merge_due()` parses live API JSON and calls external merge tooling inline, so a parser/API drift can disable merging for days without `make test` noticing. |
| C3 | critical | The deploy must have a steady-state merger or timer enabled; an absent `twiceshy-drain-merge.sh`, missing timer, or disabled timer must alert. | Add a script if still desired, then test `internal/ops`: `TestDrainMergeTimerEnabledOrAlarmed`; fake `systemctl is-enabled twiceshy-drain-merge.timer` and open validate PRs. Until the script exists, add a deployment doctor test for `scripts/twiceshy-validate.timer` plus the documented external merge service. | This checkout has `scripts/scheduled-validate.sh` but no `twiceshy-drain-merge.sh` or `twiceshy-growth-watchdog.sh`. Nothing in `make test` asserts that a production timer is enabled or that open validate PRs drain. |
| C4 | critical | Promote/adapt runs must write only non-colliding paths: record files and per-run manifests; fixed operational state must stay untracked or outside the PR diff. | `internal/ops`: `TestScheduledValidateDoesNotCommitFixedRunsState`; fake `twiceshy promote/adapt` writes `runs/promote.holds.json`, `runs/run-...-promote.json`, and a record. Assert the script stages only allowed paths after the planned fix to replace `git add -A`. | Current script uses `git add -A`, so this exact ADR-0027 failure mode is plausible. `cmd/twiceshy/holds_test.go` checks the ledger file exists, not that validate PRs avoid committing it. |
| C5 | high | Alert functions in every shell script must send `Authorization: Bearer $NTFY_TOKEN` when a token is set. | `internal/ops`: `TestShellNotifyCarriesBearerToken`; parameterize over `scripts/scheduled-import.sh`, `scripts/scheduled-validate.sh`, `scripts/corpus-stall-alarm.sh`, `scripts/sync-corpus-to-nas.sh`, and `scripts/twiceshy-watchdog.sh`. Source or execute each notify seam with fake `curl` that captures args; assert header is present with token and absent when token empty. | `scripts/corpus-stall-alarm.test.sh` stubs `notify()`, so it cannot catch missing curl headers. Go `internal/notify` now has `TestHTTPNotifier_AuthorizationHeader`, but shell alert paths are separate implementations. |
| C6 | high | Growth-stall detection must page when validated count or total corpus count does not grow over the configured window. | Add `scripts/twiceshy-growth-watchdog.sh` if it is a desired production contract, then `internal/ops`: `TestGrowthWatchdogAlertsOnNonGrowingCorpus`; fake git history/count snapshots and ntfy. | No growth watchdog script exists in this checkout. `corpus-stall-alarm.sh` detects open/red PRs, not the more dangerous case where jobs are green but validated count is flat because the merge timer is disabled or promote is doing no work. |
| C7 | high | Importers must route id allocation through the merge-safe nextid path and fail if a stale id would collide with main. | `internal/ops`: `TestScheduledImportUsesNextIDAgainstCorpusRoot`; fake `twiceshy ingest` should be invoked with `-corpus "$REPO"` and a preflight merge check should run before PR creation. | `scripts/scheduled-import.sh` delegates to `twiceshy ingest`, but shell tests do not assert the arguments or that preflight includes merge-time uniqueness, so a future script edit could bypass the safe allocator. |
| C8 | medium | Shell tests must be part of the project gate. | Add `make test-scripts` and include it in `make test` or `make ci`; test names are the Go ops tests above plus existing `.test.sh` wrappers. | Existing `.test.sh` files are not invoked by the Makefile. They can rot while `make ci` stays green. |

## D. Alerting and observability

| Priority | Severity | Missing invariant | Exact test to add | Why current tests miss it |
| --- | --- | --- | --- | --- |
| D1 | critical | Alert delivery means a request accepted by a deny-all ntfy-like endpoint, not merely an attempted POST. | `internal/notify`: `TestHTTPNotifier_DeliversWithBearerToDenyAllServer`; httptest server returns 403 unless `Authorization == "Bearer token"`. Assert token case logs no warning and no-token case logs `status=403`. | `TestHTTPNotifier_AuthorizationHeader` checks the header, and `TestHTTPNotifier_Non2xxIsLoggedNotFatal` checks logging, but no test combines auth with a deny-all endpoint to model #0093. |
| D2 | critical | Every shell alert path must authenticate and observable failures must be logged. | `internal/ops`: `TestShellNotifyDenyAllEndpoint`; fake curl returns 403 unless Bearer header exists. Assert scripts still continue, but captured log/notification status records the failure in stderr or logger. | Shell `notify()` functions use `|| true`, which is correct for control flow but dangerous for observability. Current shell tests stub the alert function and cannot see curl failures. |
| D3 | critical | A left-open red PR triggers a notification immediately and periodically. | `internal/ops`: `TestCorpusStallAlarmOpenRedPRNotifiesWithAuth`; execute `scripts/corpus-stall-alarm.sh` with fake `list_pipeline_prs` or fake Forgejo API and fake curl. Assert rc=1, body names the PR, and Bearer header is present. | `scripts/corpus-stall-alarm.test.sh` asserts red PR detection but replaces `notify()`, so auth and delivery are untested. |
| D4 | high | A non-growing corpus triggers a notification even when there are no red/open PRs. | `internal/ops`: `TestGrowthStallAlarmNoValidatedGrowth`; once growth watchdog exists, fake two snapshots with same validated count and assert ntfy. | Current stall alarm clears its state when no open PRs exist. That misses disabled validate timers, no-op promote runs, and a dead merge service that has no visible backlog yet. |
| D5 | high | Guardrail alert events must be semantically complete and carry run identifiers. | `internal/run`: extend `TestPromoteCorpus_AlertsOnAnomaly`, `TestPromoteCorpus_AlertsOnEmergencyStop`, and `TestAdaptCorpus_AlertsOnAnomaly` or add `TestRunAlertsIncludeRunContext`; assert event name, record counts, and run id in message. | Existing `internal/run/notify_test.go` only checks that an event name fired. It would pass if the alert body were useless to an operator. |
| D6 | medium | The structured logging test must be deterministic under full `-race ./...`. | Replace `internal/server/logging_test.go:TestStructuredLoggingEmitsSafeFields` with a per-request captured logger or polling parser: wait until both expected log records arrive before inspecting, close the MCP session/server cleanly, and avoid racing `bytes.Buffer` writes. Add `TestStructuredLoggingConcurrentRequestsSafe` with a mutex-backed buffer or `slogtest` handler. | #0092 says the current test flakes only under the full race suite. It writes logs from HTTP/MCP goroutines into a shared `bytes.Buffer` and then reads it without a synchronization boundary. |

5. **Shell ntfy auth matrix** (`TestShellNotifyCarriesBearerToken`, `TestShellNotifyDenyAllEndpoint`). Effort: S-M. Catches: 3.
6. **Growth/stall watchdog for non-growing validated corpus** (`TestGrowthWatchdogAlertsOnNonGrowingCorpus`). Effort: M. Catches: 2.

9. **Lifecycle state-machine matrix** (`TestLifecycleStateMachineTransitions`). Effort: M. Catches future promote/adapt/repromote desyncs; reinforces failure class 5.
10. **Add `test-scripts` and critical coverage gates to `make ci`**. Effort: S-M. Catches: makes 2, 3, and 4 visible in the normal gate; reduces the chance that 1 regresses unnoticed.
