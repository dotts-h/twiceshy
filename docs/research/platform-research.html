§3Scaling shape: one index, many runners

The owner's instinct, split in two. Query path: one service, one index, forever. Ecosystem is a WHERE clause; splitting per language breaks cross-ecosystem retrieval (a Postgres trap fires in Go and Python repos — the generic fingerprint exists for exactly this) and multiplies ops for a corpus that fits in page cache. Zoekt shards by size (~100 MiB), never language; twiceshy's whole corpus is below one shard. Runner path: per-ecosystem workers pulling from a queue — the Buildkite model: outbound-only polling, no inbound holes, scaled with compose --scale runner-python=3.

Orchestration verdict: plain Docker Compose + Caddy. k3s idles at ~1.6 GB RAM and a slice of a core (k3s's own profiling) to orchestrate one binary and one file — negative value at one node. Scale GOMAXPROCS-internally before container-count-externally.

Cloud path (post-Hetzner-increase, verified): Hetzner CX23 (2 vCPU/4 GB) €3.99/mo · CAX11 ARM €4.49 · Fly shared-cpu-1x ≈ $2/mo · Cloudflare Containers (GA Apr 2026) only interesting for scale-to-zero runners. Architecture change to migrate: zero.

Conflicting FTS5 anecdotes → measure ours. One 2026 paper claims ~1.2 s FTS latency at 5K records (likely their whole pipeline); a PHP-stack benchmark did 840 req/s on 7K docs. Resolution: a latency-budget benchmark in make ci (FTS5 MATCH + fingerprint p99 on a 1K-record corpus) instead of trusting either.

§4API surface: no new RPCs — but be stateless now

No gRPC. MCP streamable HTTP (pull) + plain HTTP (hook push) covers everything; the runner protocol is "poll a queue"; the queue is a SQLite table. The dependency budget survives intact.

Time-sensitive: the MCP 2026-07-28 release candidate removes sessions entirely. Stable today is rev 2025-11-25; the RC (2026-05-21) removes Mcp-Session-Id and the protocol session (SEP-2567) and the initialize handshake (SEP-2575). Sessions are already optional in 2025-11-25 — so: never issue a session ID, return plain JSON, 405 the GET/SSE leg, validate Origin (403 — DNS-rebinding defense), bind the tailnet interface, tokens only in Authorization headers. Any session-shaped work done now is throwaway by spec decree.

Keys & limits

format tsy_live_<base62-178bit>_<crc32> — greppable prefix, offline checksum (GitHub's rationale) at rest SHA-256, not bcrypt — ≥128-bit random keys don't need a slow KDF; constant-time compare columns scopes (pull | push | record) · created_at · expires_at · last_used_at · revoked_at — revoke = supersede limits x/time/rate token bucket per key in-process (Cloudflare free tier: 1 rule, IP-only — edge is a DDoS backstop) + SQLite daily quota (durable across restarts) · start: pull 2 rps/burst 10 · push 1 rps/burst 5 · ≤4 concurrent fail mode hook fails open (1–2 s timeout → inject nothing, agent never blocks) · server fails closed (401/429 enforced)

§7Cookbook: separate repo — the plugin marketplace is the loose link

dotts-h/Cookbook is a Claude Code plugin-marketplace repo (per the owner's settings: extraKnownMarketplaces.cookbook-dev → github:dotts-h/cookbook). That settles the topology question:

  • Keep it separate. The discriminators are access control and licensing: Cookbook is private, the engine is public AGPL, and ADR-0002 §4 already puts non-engine content outside the AGPL boundary. Submodules disqualified (a private submodule breaks every public clone, plus the documented pain); subtree/merge would bake private ops history into public AGPL history irreversibly.
  • Graceful degradation comes free: the marketplace mechanism loads the plugin when reachable and enabled; when absent, nothing breaks. No vendoring, no discovery code, no CI coupling.
  • Distribution insight (extends issue #2): ship the Phase 2 client as a twiceshy Claude Code plugin — hook config + MCP server entry (+ later a skill). Onboarding becomes: add marketplace → enable plugin → paste API key. This is also the natural index-channel vehicle (ADR-0001 §5).
  • Content flow rule: engine-generic lessons migrate individually into experience/ as records; private infra runbooks stay in Cookbook; deploy-time snapshots, if ever needed, ship as release tarballs or ORAS OCI artifacts — never vendored.

§Verification

Two adversarial verifier agents re-checked the 22 load-bearing claims against primary sources (2026-06-12): 20 confirmed exactly — Polar fees to the cent; Modal's $30/mo credit confirmed recurring; Fly/E2B pricing exact; all four arXiv IDs, task counts and finding directions; OWASP ASI06; Cloudflare free-tier rate-limit table; Tailscale Funnel constraints; Claude Code --header syntax verbatim; WAL semantics incl. checkpoint starvation.

  • corrected Stripe Managed Payments: 3.5% on top of standard Stripe fees (≈6.4% + $0.30 effective) — not the researched "5% + $0.50"; it is not fee-parity with Polar/Paddle.
  • corrected Hetzner: increase up to ~37% (not 30–35%); cheapest 2 vCPU/4 GB is CX23 €3.99 (x86) / CAX11 €4.49 (ARM); second dedicated-server adjustment 2026-06-15.
  • precision initialize-handshake removal is SEP-2575 (sessions = SEP-2567) · Firecracker's ≤5 MiB is the 1 vCPU/128 MiB reference config · Tunnel capped at 1,000/account · static bearers legal because MCP auth is optional · "Auto Dream" details are community-reported, not official docs.
  • soft spots the memory-on/off coding benchmark is vendor-run on one system · the contradicting FTS5 latency paper was not reproduced (mitigated by roadmap delta #4) · Lemon Squeezy signup status unverifiable (moot) · Cookbook contents not inspected (repo outside session scope; topology rests on its plugin-marketplace role, established by the owner's settings).
