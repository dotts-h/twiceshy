twiceshyonce bitten, twice shy — how it works

A self-hosted service that feeds hard-won engineering experience — traps, dead-ends, root causes, validated fixes — to LLM coding agents at decision time, so they stop repeating known mistakes on autopilot. A private, curated, validated StackOverflow that injects itself at the right moment.

1 · The big picture

Everything reduces to one loop: humans (and later, agents) write down what bit them; twiceshy indexes it; the next agent that walks toward the same hole gets the lesson pushed into its context before it falls in. The whole system state is one git repo + one SQLite file.

Experience records source of truth Markdown files in git: experience/YYYY/NNNN-slug.md — YAML frontmatter (machine-readable) + narrative body (where the learning lives). Diffable, reviewable, blame-able.
▼  parse + validate (internal/record)
Derived SQLite index rebuildable, never authoritative One file: a records table (full markdown for retrieval), a fingerprints table (exact-match surface), an applies_to table (stack filter), and an FTS5 full-text table (BM25). Delete it any time; twiceshy index rebuilds it from the records.
▼  retrieval (internal/index)
Search pipeline embedding-free hot path Fingerprint-exact first, BM25 lexical second; hard cap k≤3; relevance floor below which nothing is returned.
▼  serving (internal/server, cmd/twiceshy)
MCP server over streamable HTTP pull channel Tools search_experience / get_experience, bearer-token auth. Claude Code connects with claude mcp add --transport http ….
▼  (Phase 2+)
Push channel — trap cards Phase 2 A Claude Code UserPromptSubmit/PreToolUse hook calls a plain HTTP endpoint and injects 1–3 short trap cards via additionalContext — only on high-confidence matches. Deterministic: never "hope the agent asks".

2 · The experience record

One file = one lesson. The frontmatter is the retrieval surface; the body is the story. Format is normative in docs/SCHEMA.md + schema/experience-record.v1.schema.json.

---
schema_version: 1
id: exp-0001
kind: trap            # trap | fix | dead-end | convention | workflow
status: validated     # quarantined | validated | stale | superseded
title: "SQLite FTS5 MATCH treats user input as query syntax — …"

symptom:              # what an agent would observe
  summary: >
    Passing a raw query string to `… WHERE t MATCH ?` throws
    fts5: syntax error as soon as the string contains punctuation…
  error_signatures:   # exact-match surface → fingerprints derived from these
    - 'fts5: syntax error near "."'

applies_to:           # OSV-style "stack fingerprint": where this is true
  - ecosystem: "Go"
    package: "modernc.org/sqlite"

resolution:
  root_cause: "MATCH's right-hand side is a query language, not a needle…"
  fix: "tokenize, double embedded quotes, wrap each token in double quotes"
  dead_ends:          # the part StackOverflow never captures
    - tried: "escaping with backslashes"
      why_it_failed: "FTS5 has no backslash escape"

guard:                # executable proof
  repro: "experience/repro/0001-fts5-raw-match.sh"   # fails pre-fix, passes post-fix
  guarding_test: "TestSearchQuoteEscapesFTS5Input"   # keeps the fix fixed

provenance:           # trust + bi-temporal validity
  source: { author: "horia", session: null, pr: null }
  recorded_at: 2026-06-12
  valid: { from: 2026-06-12, until: null }
  superseded_by: null  # supersede, never delete
---

Lifecycle: supersede, never delete

quarantined — sandbox fail-to-pass + human PR review → validated → stale or superseded

Agent-proposed records are born quarantined and never enter the push channel (memory-poisoning defense: <0.1% poisoned store ⇒ ≥80% attack success in the literature). Promotion is a pull request — git is the trust boundary. A record never disappears: its validity interval is closed and superseded_by links the replacement.

3 · Retrieval: precedence, cap, floor

When a query arrives (an error message, a symptom description), the pipeline runs in strict precedence order:

stagehowwhy first/second
1Fingerprint-exact The query text is normalized (lowercased; UUIDs/addresses/hex/paths/standalone numbers replaced by placeholders) and hashed: sha256("generic\n" + normalized). Hash equality against fingerprints derived from every record's error_signatures at index time. Deterministic, free, zero false positives — covers "agent looping on a known bug". Two scopes per signature: app (repo-specific) and generic (cross-repo).
2Lexical (BM25/FTS5) The query is tokenized and each token is double-quoted before reaching MATCH (untrusted input is FTS5 syntax otherwise — that's literally record exp-0001), joined with OR, ranked by bm25(). Error corpora are identifier-heavy; dense embeddings fail silently on exact identifiers. Note: SQLite's bm25() is negative, lower-is-better — the sign is flipped exactly once at the API boundary (record exp-0002).
3Dense later sqlite-vec + RRF(k=60) fusion, pull-channel only, behind an embedding cache. Deliberately absent from the hot path — see invariants.

Both stages are filtered by status (validated only, unless a pull-channel caller explicitly asks for labeled quarantined records) and optionally by the stack fingerprint (ecosystem/package against applies_to).

Invariant — k ≤ 3. Never more than three results, ranked. Context is a scarce, fragile resource; more retrieval measurably hurts.
Invariant — relevance floor. Below the floor, nothing is returned for injection. Near-miss injection (related-but-wrong record) is the #1 failure mode: one plausible-but-wrong document hurts agents more than random noise. Returning nothing is a feature.
Invariant — embedding-free hot path. The hook-lookup path is fingerprint + lexical only, structurally (separate code paths), so ~1000 agents can poll it continuously on a NAS.

4 · Serving: the three channels

Pull — MCP tools built

search_experience (query + optional ecosystem/package filter → ≤3 hits with ids, scores, match kind) and get_experience (id → full record markdown). Streamable HTTP — the deprecated HTTP+SSE transport is itself record exp-0003. Constant-time bearer auth; no unauthenticated mode.

Push — hooks Phase 2

Claude Code hooks call a plain HTTP endpoint per turn; high-confidence matches come back as 1–3 trap cards injected via additionalContext. This is what interrupts autopilot at the moment it matters — the evidence says models don't call memory tools unprompted.

Index — ambient awareness Phase 2+

Generated skill / AGENTS.md one-liners so the model knows the store exists and what categories it holds.

Running it

# rebuild the derived index from the records
twiceshy index -corpus . -db twiceshy.db

# serve the MCP pull channel (rebuilds the index on start)
TWICESHY_TOKEN=… twiceshy serve -corpus . -db twiceshy.db -addr :8722

# connect Claude Code
claude mcp add --transport http twiceshy https://nas.tailnet.ts.net/ \
  --header "Authorization: Bearer …"

5 · Keeping it honest: doctors & evals Phases 4–5

doctorjob
D1 dedup/reconcilenew candidate vs top-k similar → LLM arbitrates ADD / UPDATE / SUPERSEDE / NOOP. Delta-updates only — never whole-store rewrites.
D2 stalenesscross-check applies_to against live versions/docs. A stale store is worse than none — it becomes the outdated-context prompt.
D3 revalidation ⭐CI for memories — re-execute every record's repro script in a sandbox on a schedule and on dependency bumps (fail-to-pass discipline). No published system does this.
D4 lifecyclereinforce helpful records, decay never-hit ones — archive, never delete (a trap hit once a year may matter most).
D5 abstractioninduce convention cards from clusters of related episodes.

Phase 5 adds the trap-avoidance eval: for each recorded trap, a task that walks an agent toward it, run memory-on vs memory-off, scoring avoidance + steps/tokens. Nobody has published such a suite.
